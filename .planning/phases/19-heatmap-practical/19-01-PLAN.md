# Plan 19-01: Automatic Click Tracking and Heatmap Backend

<objective>
Add browser-side automatic click tracking on diagram nodes. Clicks are batched client-side every 2 seconds and flushed to a new POST endpoint. The server stores frequency data in-memory per file. GET /api/heatmap/:file merges click data with session data. This enables the heatmap to show useful data from the moment users start clicking -- no MCP session setup required.
</objective>

<tasks>

## Task 1: Create HeatmapStore and heatmap-routes.ts

**File:** src/server/heatmap-routes.ts (new)

Create an in-memory HeatmapStore class:
- `private data: Map<string, Map<string, number>>` -- file -> (nodeId -> count)
- `increment(file, counts: Record<string, number>)` -- merge counts into store
- `getCounts(file): Record<string, number>` -- return counts for a file
- `clear(file)` -- clear counts for a file (for testing)

Create `registerHeatmapRoutes(routes, heatmapStore, sessionStore)`:
- `GET /api/heatmap/:file` -- merge heatmapStore.getCounts + sessionStore.getHeatmapData, return merged object
- `POST /api/heatmap/:file/increment` -- read JSON body `{counts: Record<string, number>}`, call heatmapStore.increment, broadcast `heatmap:update` via wsManager

Export the HeatmapStore class.

## Task 4: Create interaction-tracker.js

**File:** static/interaction-tracker.js (new)

Create SmartBInteractionTracker IIFE:
- Internal state: `pendingCounts = {}`, `flushTimer = null`, `FLUSH_INTERVAL = 2000`
- `trackClick(nodeId)`: increment `pendingCounts[nodeId]`
- `flush()`: if pendingCounts is empty, return. POST `/api/heatmap/:file/increment` with `{counts: pendingCounts}`. Reset pendingCounts.
- `startFlushing()`: setInterval(flush, FLUSH_INTERVAL)
- `resetForFile()`: flush current, clear pendingCounts
- `init()`: start flushing, listen to `selection:changed` event on SmartBEventBus for node clicks

Public API: `{ init, trackClick, flush, resetForFile }`

**Commit:** feat(19-01): create interaction-tracker.js for client-side click batching
